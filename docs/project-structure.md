# Описание структуры проекта (FSD — Feature-Sliced Design)

Документ описывает, как организовать фронтенд-код проекта по методологии FSD (Feature-Sliced Design) и как текущие папки/файлы проекта соответствуют этой структуре.

**Коротко о FSD**
- FSD делит код по уровням (app, pages, widgets, features, entities, shared), а не по типу (components, stores и т.д.).
- Это помогает масштабировать приложение, упрощает навигацию и реиспользование.

**Рекомендуемая структура (в `resources/js`)**

- `app/` — точка входа приложения, глобальные провайдеры, роутинг и конфигурация.
  - `app/entry.ts` — монтаж Vue + Inertia и глобальные плагины.
  - `app/providers/*` — провайдеры (i18n, store, theme и т.п.).

- `pages/` — страницы (роуты приложения). Каждая страница — отдельная папка с компонентами, используемыми только на ней.
  - `pages/HomePage/index.vue`
  - `pages/UsersPage/index.vue`

- `widgets/` — составные UI-блоки, которые объединяют features/entities для отображения сложных участков интерфейса.
  - `widgets/UserListWidget/` — widget может использоваться на нескольких страницах.

- `features/` — бизнес-функции: функциональные куски поведения, которые могут использоваться в разных виджетах и страницах.
  - `features/auth/` — логика логина/регистрации, формы и взаимодействия с API.

- `entities/` — модели предметной области + их UI-части (карточки, сущности, CRUD-виды).
  - `entities/user/` — `ui/`, `model/`, `api/` внутри папки entity.

- `shared/` — общие утилиты, UI-кит проекта и низкоуровневые компоненты.
  - `shared/ui/` — кнопки, инпуты, тейкхелперы по стилю (элементы, заведомо переиспользуемые).
  - `shared/lib/` — утилиты, api-клиенты, типы.

**Как это соотносится с текущим репозиторием**
- Текущая точка входа: `resources/js/app.ts` — здесь рекомендуется создать `app/entry.ts` и переносить туда инициализацию Vue/Inertia.
- Сейчас есть `resources/js/Pages/` — соответствует уровню `pages/` в FSD.
- `resources/js` — можно привести к корню FSD: создать папки `app`, `pages`, `widgets`, `features`, `entities`, `shared` и постепенно мигрировать компоненты.

**Примерный план миграции**
1. Создать `resources/js/app/entry.ts`, перенести туда код из `resources/js/app.ts`.
2. Создать `resources/js/shared/ui/Button.vue` и `resources/js/shared/ui/Input.vue` — заменить все локальные кнопки.
3. Выделить сущности (`entities/user`) и переместить карточки/модели.
4. Выделять `features` по мере роста логики (auth, profile, search и т.д.).

**Правила именования и импорта**
- Папка и файл компонента: `ComponentName/index.vue` или `ComponentName/ComponentName.vue`.
- По возможности использовать абсолютные алиасы: `@/shared/ui/Button` (настроить `tsconfig.json`/`vite` для алиасов).
- Компоненты и модули внутри feature/entity инкапсулируются и экспортируют минимально необходимое API.

**Интеграция с Vue + Inertia + TypeScript**
- Точка входа (например, `app/entry.ts`) должна:
  - Инициализировать `createApp` или `createInertiaApp`.
  - Регистрировать глобальные провайдеры и `shared`-плагины.
- Типы для страниц и глобальных свойств держать в `shared/lib/types`.

**Пример структуры файлов (в виде дерева)**

```
resources/js/
├─ app/
│  ├─ entry.ts
│  └─ providers/
├─ pages/
│  └─ HomePage/
├─ widgets/
│  └─ UserListWidget/
├─ features/
│  └─ auth/
├─ entities/
│  └─ user/
└─ shared/
   ├─ ui/
   └─ lib/
```

Если хотите, могу:
- Сгенерировать начальную структуру (создать папки и шаблонные файлы `entry.ts`, `shared/ui/Button.vue` и т.д.).
- Настроить алиасы в `tsconfig.json` и `vite.config.js`.
- Установить конкретный UI-kit — укажите точное название пакета (`schd-ui-kit` или, возможно, вы имели в виду другой).
